# 6A. MARKET DATA + FX VALUATION PROTOCOL

This section is mandatory.

## Primary rule
After the U.S. regular session has ended, actively attempt a fresh same-day valuation update.
Carry-forward is fallback, not baseline.

## Machine-readable audit preference
If a pricing audit exists in `output/pricing/` for the requested close date, read it first and treat it as the preferred operational pricing input for this run.

Use the latest matching audit to resolve, where available:
- holdings freshness labels
- unresolved holdings
- source tiers used
- FX basis used
- holdings coverage by count
- invested-weight coverage
- carry-forward versus update decision

Only rerun or redo the pricing pass manually if the audit is:
- missing
- for the wrong close date
- internally inconsistent
- too incomplete for the current run
- or clearly stale relative to the requested close date

A prior pricing audit is preferred over ad hoc repeated retrieval for the same run date, but it must not be treated as current if it does not match the requested close date.

## Allowed ETF close hierarchy
Per holding, use:
1. official exchange or primary market close
2. issuer or fund provider page if it clearly shows latest market price or NAV-relevant trading price
3. major market data source with latest official close
4. reputable financial media quote page
5. prior verified close from the latest stored report only if no fresh close can be verified

## Allowed FX hierarchy for USD-to-EUR conversion
1. same-day market close EUR/USD snapshot from a reputable FX, central-bank, or market-data source
2. latest official ECB or equivalent reference rate if same-day close is not yet reliably available
3. prior verified EUR/USD from the latest stored report only if no fresh FX reference can be verified

## Per-ticker rule
Attempt a fresh close for every held ETF separately.
Do not require one single source for all holdings.

## Mandatory internal pricing coverage table
Build an internal table with at minimum:
- Ticker
- Previous price
- Fresh price found? Yes/No
- Source tier used
- FX basis used
- Status = Fresh close / Fresh fallback source / Carried forward

## Incomplete-set rule
If at least 75% of holdings by count OR at least 85% of invested portfolio weight has a fresh verifiable close:
- update those holdings to fresh close
- carry forward only the unresolved holdings
- still update total NAV and equity curve

If less than that threshold is met:
- carry forward the full prior verified portfolio valuation
- explicitly state that the mark-to-market was not updated because the fresh close-set was too incomplete

## Anti-freeze rule
Do not leave the equity curve unchanged merely because one or two holdings or the exact same-day FX close could not be perfectly verified.

# 7. OPEN DISCOVERY + DYNAMIC LANE FRAMEWORK

Every run must begin with a broad internal discovery sweep before deciding which lanes appear in the published report.

## Core design rule
Do **not** use a small fixed lane list as the front-end discovery gate.
Use broad open discovery first, then create and rank dynamic lanes.

## Broad discovery domains
Each run must scan across broad investable domains, including where relevant:
- macro / cyclical leadership
- policy / rates / duration sensitivity
- geopolitics / sovereign resilience
- supply chain / logistics / chokepoints
- food security / agriculture inputs / farm productivity
- water security / treatment / leakage reduction / irrigation
- power / grid / electrification
- advanced energy / nuclear / fuel systems
- digital infrastructure / semiconductors / cyber / networking
- industrial automation / robotics / productivity capex
- critical minerals / refining / recycling / enabling materials
- healthcare innovation / biotech / diagnostics / therapeutic platforms
- financial infrastructure / payments / market plumbing / tokenization
- regional industrialization / reshoring / country-level capex winners
- real assets / infrastructure materials / engineering beneficiaries

This list is a **broad discovery taxonomy**, not a publication checklist.
The system may surface categories outside these labels if clearly investable and relevant.

## Candidate-lane construction
After the discovery sweep, construct 8–15 candidate lanes.
A lane is an investable basket or category that represents a coherent macro, geopolitical, structural, or second-order opportunity.

Examples of lanes:
- AI compute infrastructure
- food security and agriculture inputs
- water infrastructure
- defense innovation
- maritime chokepoints and shipping infrastructure
- obesity / metabolic-health ecosystem
- India industrialization beneficiaries

## Candidate-lane quality rule
A lane must be:
- investable through an ETF or a clearly definable basket
- relevant to the 3–12 month horizon
- sufficiently differentiated from other candidates
- connected to current macro, geopolitical, or structural drivers

## Stable taxonomy rule
Every candidate lane must be tagged internally into a persistent taxonomy for:
- continuity across runs
- anti-omission checks
- watchlist memory
- historical comparison

The taxonomy is a back-end memory layer, not a front-end gate.

## Mandatory challenger rule
Each weekly run must include at least 3 challenger lanes that were not all published in the prior run.
At least 1 challenger lane should come from outside the prior published radar if evidence supports it.

## Internal-only candidate-lane table
Before publication, build an internal table containing at minimum:
- Lane
- Taxonomy tag
- Primary ETF / vehicle
- Alternative ETF / vehicle
- Structural Strength score
- Macro Alignment score
- Second-Order Relevance score
- Implementation Quality score
- Valuation / Crowding score
- Timing Confirmation score
- Portfolio Differentiation score
- 3–12 month Persistence score
- Prior-run status
- Publish? Yes/No

Do not print the full candidate table unless explicitly asked.
Use it to drive the ranking and publication decision.

## Two-layer lane scoring
Score every lane from 1 to 5 on:

### Layer 1 — Structural / strategic quality
- Structural Strength
- Persistence of tailwind
- Market access / implementation quality

### Layer 2 — Current regime fit
- Macro Alignment
- Second-Order Relevance
- Timing Confirmation
- Valuation / Crowding Balance
- Portfolio Differentiation

### Suggested weighting
Lane Score =
- Structural Strength × 0.18
- Persistence × 0.12
- Implementation Quality × 0.12
- Macro Alignment × 0.18
- Second-Order Relevance × 0.12
- Timing Confirmation × 0.12
- Valuation / Crowding × 0.08
- Portfolio Differentiation × 0.08

You may adjust interpretation at the margin, but keep the framework stable.

## Publication rule
The published Structural Opportunity Radar should usually contain:
- 5 lanes by default
- allowed range: 5 to 8 lanes

Only publish the highest-ranked lanes.
Compression happens **after** discovery, not before it.

## Anti-omission rule
Also publish 1 to 3 `Best structural opportunities not yet actionable` if strong structural candidates score well on structural strength but do not yet justify full inclusion in the live radar.

## Freshness + continuity rule
Each run must evaluate:
- which published lanes remain
- which lanes enter
- which lanes drop out
- which lanes remain near-miss challengers
- what changed in timing, urgency, or conviction

The report should explain lane change elegantly, but not become a process log.
